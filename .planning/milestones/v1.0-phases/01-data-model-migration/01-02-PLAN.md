## Goal

Wire door hardware seed data into DEFAULT_SETTINGS, implement v2-to-v3 schema migration (additive -- preserves all existing user data), update CURRENT_SCHEMA_VERSION, fix all existing test factories and verify-calc script to include the new `doorHardware` field, and add `doorHardware` case to use-settings.ts getUsageCount. After this plan, the entire app compiles clean, all existing tests pass, the verification script passes, and migration correctly upgrades v1 and v2 data to v3.

## Context for Executor

**Codebase patterns to follow:**
- Migration: `migrateState()` in `src/storage/storage-service.ts` -- currently handles v1->v2 in a single pass. Refactor to sequential version checks.
- Default state: `createDefaultAppState()` in `src/data/index.ts` -- returns `{ schemaVersion: N, projects: [], settings: DEFAULT_SETTINGS }`
- Test factories: `baseLineItem()` / `validItem()` inline in test files -- must include all required `LineItem` fields
- Storage tests: `src/storage/storage-service.test.ts` uses `// @vitest-environment jsdom` pragma

**User decisions (LOCKED):**
- v2->v3 migration is ADDITIVE -- existing settings (glass types, labor rates, etc.) MUST be preserved. Only ADD `doorHardware` catalog.
- Existing line items gain `doorHardware: []` (empty array) during migration -- no auto-population of defaults during migration.

**Key pitfalls from research:**
- Migration must handle v1->v3 jump (apply v1->v2 then v2->v3 sequentially)
- Settings migration must MERGE, not replace -- spread existing settings, overlay only `doorHardware`
- Every `LineItem` factory in tests and scripts must get `doorHardware: []`

**Step 3: Refactor migrateState() for sequential version upgrades**

Replace the `migrateState` function in `src/storage/storage-service.ts` with:

```typescript
function migrateState(parsed: Partial<AppState>): AppState {
  const defaults = createDefaultAppState()
  let version = parsed.schemaVersion ?? 1
  let projects = parsed.projects ?? []
  let settings = parsed.settings ?? defaults.settings

  // v1->v2: Replace all settings with new seed data (B-007).
  // Settings are incompatible: SystemType gained laborMode, FrameSystem lost laborHoursPerUnit.
  if (version < 2) {
    settings = defaults.settings
    projects = projects.map(p => ({
      ...p,
      lineItems: (p.lineItems ?? []).map((li: Partial<LineItem>) => ({
        ...li,
        manHours: 0,
        conditionIds: [],
      })),
    }))
    version = 2
  }

  // v2->v3: Add door hardware (additive, preserves existing settings)
  if (version < 3) {
    settings = {
      ...settings,
      doorHardware: defaults.settings.doorHardware,
    }
    projects = projects.map(p => ({
      ...p,
      lineItems: (p.lineItems ?? []).map((li: any) => ({
        ...li,
        doorHardware: li.doorHardware ?? [],
      })),
    }))
    version = 3
  }

  const migrated: AppState = {
    schemaVersion: CURRENT_SCHEMA_VERSION,
    settings: settings as AppState['settings'],
    projects: projects as AppState['projects'],
  }

  saveAppState(migrated)
  return migrated
}
```
